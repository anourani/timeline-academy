# PRD: AI Mode — new timelines default to Small scale

> **Status:** Draft
> **Owner:** Timeline Academy
> **Last updated:** 2026-04-20

---

## 1. Context / Background

Timeline Academy's **AI mode** lets a user type a subject (e.g. "Frida Kahlo") and receive a pre-populated timeline with title, description, events, and categories generated by the LLM. Full product background lives in [`plan.md`](./plan.md) — the "Enter a name. we build the timeline." flow.

Each timeline has a **scale** setting — `'large' | 'small'` — that controls how wide each month/quarter is in the rendered timeline. The values are defined in `src/constants/scales.ts`:

| Scale | `monthWidth` | `quarterWidth` |
|------:|:------------:|:--------------:|
| large |      28      |       7        |
| small |      20      |       5        |

Users can switch between the two from the Settings panel via `src/components/Settings/ScaleSelector.tsx`. The chosen scale is persisted on the `timelines.scale` column in Supabase for logged-in users and in the local draft blob for logged-out users.

## 2. Problem statement

Today, when a user proceeds through AI mode — logged in or logged out — the newly created timeline opens in the editor with scale set to **Large**. The product intent is for AI-generated timelines to open at **Small** so they feel compact and scannable on first view, matching the density the rest of the product defaults to.

### Root cause (from codebase investigation)

`loadTimeline('new')` in `src/hooks/useTimeline.ts:104-108` inserts the new Supabase row **without specifying `scale`**, so the Postgres column default is applied. That column default was `'large'` from the original migration (`supabase/migrations/20250203181722_small_prism.sql:17`) until it was flipped to `'small'` by `supabase/migrations/20260420000000_default_timeline_scale_small.sql`. On any environment without that migration deployed — or during the ~2s window before autosave overwrites the DB with the UI's hardcoded `'small'` — a hard reload reads back `'large'` and the UI flips to Large.

## 3. Goals

- New timelines created via AI mode consistently open at `scale: 'small'` on first render.
- The behavior is stable after a hard page reload — i.e. the persisted record is `'small'`, not the DB column default.
- Both the logged-in (Supabase) and logged-out (localStorage draft) paths behave the same way.

## 4. Non-goals

- Changing the scale of **existing** timelines. Anything already persisted keeps its stored scale.
- Changing defaults for the manual ("Create my own timeline") or CSV-import creation flows — they already default to Small and should not regress.
- Introducing a user preference for default scale. That's a separate, future feature.

## 5. User stories

1. As a **logged-in user** who opens AI mode and submits a subject, I arrive in the editor with the scale toggle showing **Small**.
2. As a **logged-out user** who opens AI mode and submits a subject, I arrive in the editor with the scale toggle showing **Small**.
3. After either of the above, if I **hard-refresh the page**, the timeline is still at **Small** — proving the persisted record, not just ephemeral UI state, is Small.

## 6. Functional requirements

- **FR1** — The first Supabase row persisted for an AI-generated timeline must have `scale = 'small'`, independent of the column default.
- **FR2** — The first localStorage draft persisted for an AI-generated timeline must have `scale = 'small'`.
- **FR3** — Re-loading the timeline from Supabase or localStorage returns `scale = 'small'` until the user explicitly changes it.
- **FR4** — No regression on other creation paths (manual, CSV import, resume draft, switch timeline).
- **FR5** — No retroactive mutation of existing timeline rows.

## 7. Technical approach

Surface area is intentionally small. The fix is one line of production code plus the already-landed migration.

### 7.1 Frontend: make the INSERT authoritative

In `src/hooks/useTimeline.ts:104-108`, pass `scale: 'small'` to the `insert(...)` call so the new row is created with Small regardless of the Postgres column default:

```ts
const { data: newTimeline, error: createError } = await supabase
  .from('timelines')
  .insert({ title: DEFAULT_TIMELINE_TITLE, user_id: user.id, scale: 'small' })
  .select('id')
  .single();
```

This closes the autosave race window (where a reload within ~2s of creation reads back the DB default) and makes the code match the hardcoded `scale: 'small'` already returned from `loadTimeline('new')` at `src/hooks/useTimeline.ts:120`.

### 7.2 Database: default already corrected

`supabase/migrations/20260420000000_default_timeline_scale_small.sql` (already authored) flips the `timelines.scale` column default from `'large'` to `'small'`. It ships as belt-and-braces; the frontend change above makes the app correct even on environments where this migration has not yet run.

### 7.3 No-change areas (verify only)

- `src/hooks/useTimelineScale.ts:5` — React state already defaults to `'small'`.
- `src/utils/draftStorage.ts:78-87` — `createDraft()` already sets `scale: 'small'` for logged-out users.
- `src/App.tsx:128-148` (logged-out AI mode) and `src/App.tsx:319-335` (logged-in AI mode) — both paths forward the scale value into `handleScaleChange`; no logic change required.
- `src/hooks/useAIMode.ts` — the AI generation response intentionally does not carry a scale; it stays that way.

## 8. Acceptance criteria

- **AC1 (logged-out AI mode)** — Fresh session, empty localStorage. Open AI mode, submit a subject. Land in the editor → Settings panel's scale toggle shows **Small**.
- **AC2 (logged-in AI mode)** — Signed in with zero existing timelines. Open AI mode, submit a subject. Land in the editor → scale toggle shows **Small**.
- **AC3 (persistence)** — Immediately after AC2, hard-refresh the page. Scale toggle still shows **Small**. The Supabase row's `scale` column is `'small'`.
- **AC4 (no regression)** — "Create my own timeline" flow and CSV import flow still default to **Small**.
- **AC5 (no retroactive mutation)** — Timelines that existed before the change retain their previously stored scale.

## 9. Verification plan

1. `npm run dev` and walk through AC1, AC2, AC3, AC4 in a browser.
2. For AC3, query Supabase directly (SQL editor or `supabase` CLI) on the newly created row, within a second of creation, to confirm `scale = 'small'` before the autosave debounce could have fired.
3. `npm run lint` passes.
4. `npm run build` passes.

## 10. Risks & rollback

- **Risk:** Hardcoding `'small'` on insert could conflict with a future "user-preferred default scale" feature. *Mitigation:* acceptable for now; revisit the insert site when that feature is designed.
- **Risk:** The DB migration has not been applied in every environment. *Mitigation:* the frontend change makes the app correct regardless of the migration state.
- **Rollback:** Revert the one-line change in `src/hooks/useTimeline.ts`. The DB default (`'small'` after the 2026-04-20 migration) continues to produce the desired behavior on its own in environments where that migration has run.
